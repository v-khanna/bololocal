# Bolo — Development Guide

How to build, test, and work on Bolo locally.

For the system design overview, see [ARCHITECTURE.md](ARCHITECTURE.md).
For signing + notarization + Setapp release, see [RELEASE.md](RELEASE.md).

---

## Prerequisites

- macOS 15 (Sequoia) or newer — runtime requirement for the app
- Apple Silicon (M1+) — the Qwen3-TTS model requires MLX
- Xcode 15+ (the project was built and verified with Xcode 26.4.1)
- Homebrew (native ARM at `/opt/homebrew`)
- `xcodegen` — `brew install xcodegen`
- Metal Toolchain — if you see `mlx.metallib` build errors, run
  `xcodebuild -downloadComponent MetalToolchain` (one-time, ~687 MB)

The project itself has two Swift Package dependencies, resolved
automatically by Xcode on first build:

- [HotKey](https://github.com/soffes/HotKey) — global keyboard shortcut
- [soniqo/speech-swift](https://github.com/soniqo/speech-swift) (package
  name `Qwen3Speech`, we use the `Qwen3TTS` product)

## First-time setup

```bash
git clone <your fork> ~/Code/bolo
cd ~/Code/bolo
xcodegen generate
```

That's it. Open `Bolo.xcodeproj` in Xcode, or build/test from the CLI
(see below). The `.xcodeproj` is generated from `project.yml` and is
gitignored — never edit it by hand; edit `project.yml` and re-run
`xcodegen generate`.

## Daily workflow

### Build

```bash
xcodebuild -scheme Bolo -destination 'platform=macOS,arch=arm64' build
```

Or in Xcode: `⌘B`.

The first build resolves SPM dependencies (~30 sec network + ~2 min
to compile MLX-Swift). Subsequent builds are incremental.

### Run

From Xcode: `⌘R`. The app launches as a menu bar item (no Dock icon
because `LSUIElement = true`). Look for the waveform glyph in the
top-right menu bar.

From CLI:

```bash
xcodebuild -scheme Bolo -derivedDataPath build \
    -destination 'platform=macOS,arch=arm64' build
open build/Build/Products/Debug/Bolo.app
```

### Test

```bash
xcodebuild -scheme Bolo -destination 'platform=macOS,arch=arm64' test
```

Or in Xcode: `⌘U`.

Expected: **23 tests pass, 1 skipped** (the heavy Qwen3 integration
test is skipped by default; see below).

### Run the heavy Qwen3 integration test

The default test pass does not load the real Qwen3 model — that would
trigger a ~500 MB download on first run and ~3 sec of model load
every time. To run the real-model test:

```bash
BOLO_RUN_HEAVY_TESTS=1 xcodebuild \
    -scheme Bolo -destination 'platform=macOS,arch=arm64' test \
    -only-testing:BoloTests/TTSEngineTests/test_qwen3Engine_synthesize_realModel
```

First run downloads weights; subsequent runs use the cache at
`~/Documents/huggingface/`.

### Clean build

```bash
rm -rf build Bolo.xcodeproj DerivedData
xcodegen generate
xcodebuild -scheme Bolo build
```

If SPM packages get into a weird state:

```bash
rm -rf ~/Library/Caches/org.swift.swiftpm
rm -rf ~/Library/Developer/Xcode/DerivedData/Bolo-*
```

## Working on the code

### Project layout

```
~/Code/bolo/
├── Bolo/                        # App target sources (see ARCHITECTURE.md §3)
├── BoloTests/                   # Unit tests
├── docs/                        # ARCHITECTURE, DEVELOPING, RELEASE, SETAPP
├── scripts/                     # build.sh, notarize.sh, release.sh
├── project.yml                  # xcodegen source of truth — edit this, not the .xcodeproj
├── README.md, SCOPE.md          # entry-level docs
├── PRIVACY.md                   # privacy policy
└── .gitignore                   # excludes Bolo.xcodeproj, Info.plist, entitlements (all generated)
```

### Adding a Swift source file

1. Drop the file under `Bolo/<category>/<Name>.swift`. The `project.yml`
   uses `sources: - path: Bolo` recursively, so new files are picked up
   on the next `xcodegen generate`.
2. Run `xcodegen generate`.
3. Build / test.

### Adding a new SPM dependency

Edit `project.yml`:

```yaml
packages:
  HotKey:
    url: https://github.com/soffes/HotKey
    from: 0.2.0
  YourPackage:                                 # add this block
    url: https://github.com/owner/repo
    from: 1.0.0

targets:
  Bolo:
    dependencies:
      - package: HotKey
      - package: Qwen3Speech
        product: Qwen3TTS
      - package: YourPackage                   # add this line (and product: ... if needed)
```

Then `xcodegen generate && xcodebuild -resolvePackageDependencies`.

### Editing entitlements or Info.plist

Both are generated by xcodegen. **Do not** edit the files on disk —
they're in `.gitignore` and will be overwritten on every regenerate.
Edit `project.yml`'s `info.properties:` and `entitlements.properties:`
blocks. Then `xcodegen generate`.

### Renaming the app

A global find-and-replace of `Bolo` and `bolo` plus directory renames.
Pattern documented in the git history (see commit `0ad844c rename:
HearIt → Bolo`). The exhaustive list:

- All `Bolo` and `bolo` references in `Bolo/`, `BoloTests/`, `docs/`,
  `scripts/`, `README.md`, `SCOPE.md`, `PRIVACY.md`, `project.yml`.
- Directory renames: `Bolo/` → `NewName/`, `BoloTests/` → `NewNameTests/`.
- File renames: `BoloApp.swift` → `NewNameApp.swift`.
- Bundle ID: `com.virkhanna.bolo` → your choice.
- Notary credential profile name: `bolo-notary` → `newname-notary`.

## Concurrency rules of thumb

Bolo compiles with `SWIFT_STRICT_CONCURRENCY: complete`. A few patterns
are used throughout — match them when adding code:

- **UI / AppKit code** → `@MainActor` on the type. AppDelegate,
  Coordinator, PlaybackController, every UI class.
- **Stateful engine code that touches non-Sendable types** → `actor`.
  ModelManager, Qwen3TTSEngine.
- **Cross-isolation closures from background to MainActor** →
  `Task { @MainActor in ... }`. Used heavily in the
  `progressHandler` callback path.
- **Conforming an actor to a `Sendable` protocol** → mark the protocol
  methods `nonisolated` and delegate to an internal `_method` on the
  actor. See `Qwen3TTSEngine.synthesize` for the canonical example.
- **External library types without Sendable annotation** →
  `@preconcurrency import LibraryName`. Used for `Qwen3TTS` and
  `ApplicationServices`.
- **Carbon / C callbacks known to fire on main** →
  `MainActor.assumeIsolated { ... }` inside the callback closure.

When in doubt, let the compiler error guide you — Swift 6's diagnostics
are detailed enough that the right fix is usually obvious.

## Debugging

### Hotkey not firing

1. Confirm Accessibility access was granted (System Settings → Privacy
   & Security → Accessibility → enable Bolo). The Carbon hotkey API
   doesn't strictly need it, but newer macOS releases enforce Input
   Monitoring for global hotkeys in some apps. Check System Settings
   → Privacy & Security → Input Monitoring as well.
2. Make sure no other app is registering ⌘⇧R. Common collisions:
   Screen recording apps, dictation tools.
3. Restart the app — Carbon hotkey registration can occasionally get
   wedged.

### AX capture returns empty

`TextCaptureManager.captureFromAccessibility` returns nil when:

- Accessibility permission isn't granted.
- The focused app doesn't expose `kAXSelectedTextAttribute` (Terminal,
  some Electron apps, web views with custom selection).

The clipboard fallback should kick in. If the clipboard is also empty,
nothing happens (logged: `Bolo: nothing selected`).

### Qwen3 model fails to load

Most likely causes:

- **Network**: first-run download requires reachability to
  `huggingface.co`. Check `curl https://huggingface.co/`.
- **Disk space**: the model + tokenizer combined are ~520 MB. Check
  `df -h ~/Documents`.
- **Metal toolchain missing**: build will fail with
  `mlx.metallib not found`. Run
  `xcodebuild -downloadComponent MetalToolchain`.
- **Cache corruption**: delete `~/Documents/huggingface/aufklarer` and
  `~/Documents/huggingface/Qwen` and retry.

### Audio plays but at wrong pitch

If `AVAudioUnitVarispeed.rate` is set incorrectly (e.g. `0` or `NaN`),
audio gets distorted. The `Speed` value type clamps to `[0.5, 2.0]` —
if you're seeing this, something's bypassing `Speed.init` and writing
directly to `varispeed.rate`. Grep for `varispeed.rate =` and verify
all writes go through `Float(speed.value)`.

### Tests pass locally but fail in CI

We don't currently have CI. If you set one up, be aware:

- The Qwen3 integration test will try to download ~500 MB on first
  run — gate it via the `BOLO_RUN_HEAVY_TESTS` env var.
- AX tests don't work in CI (no focused app, no permission).
- Audio tests don't work in headless CI (no output device).

## Where things live in memory

If you're debugging memory growth:

```
xcrun instruments -t Allocations Bolo.app
```

Expected allocations after first ⌘⇧R:

- Qwen3 model arrays (MLX backend): ~400 MB
- Tokenizer + ancillary state: ~50 MB
- Audio buffers: brief peaks of ~10 MB per synthesize call
- SwiftUI / AppKit state: ~10 MB

After 5 minutes idle, the Qwen3 arrays should be released — total
allocation drops back to under 50 MB.

If the model isn't unloading, check:

- Is `ModelManager.idleTask` getting cancelled by repeated touches?
  Add a print in `resetIdleTimer` to confirm timing.
- Is there a strong reference to the model held outside ModelManager?
  Search for any code path that captures the model returned by
  `ensureLoaded()` in a long-lived closure.

## Submitting changes

1. Branch off main.
2. `xcodegen generate && xcodebuild test` must pass (23 tests, 1 skipped).
3. Match the existing code style — focused files, one responsibility
   per file, named clearly, no comments unless the *why* is non-obvious.
4. Commit messages follow conventional-commits style (see git log):
   `feat(area): ...`, `fix(area): ...`, `chore: ...`, `build: ...`,
   `refactor: ...`, `docs: ...`, `rename: ...`.
5. PR description should explain *why*, not *what* (the diff already
   tells you what).
